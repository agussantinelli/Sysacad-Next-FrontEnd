<h2>рҹҺҜ El DesafГӯo (Legacy vs. Next)</h2>

<ul>
    <li><strong>Lentitud crГӯtica:</strong> Colapsos sistГ©micos en periodos de alta demanda, especialmente durante las fechas de inscripciГіn.</li>
    <li><strong>UX obsoleta:</strong> Interfaces rГӯgidas no optimizadas para el uso cotidiano en dispositivos mГіviles.</li>
    <li><strong>FragmentaciГіn:</strong> DispersiГіn de la informaciГіn acadГ©mica que dificulta el seguimiento real del progreso del alumno.</li>
    <li><strong>Inestabilidad del Servidor:</strong> CaГӯdas constantes que interrumpen procesos vitales, generando un clima de frustraciГіn constante.</li>
    <li><strong>Malestar Estudiantil:</strong> Las fallas recurrentes provocan quejas legГӯtimas de los alumnos y del Centro de Estudiantes, derivando en un descontento generalizado hacia la instituciГіn.</li>

<p>Esta reingenierГӯa transforma radicalmente esa experiencia, proponiendo una plataforma <strong>reactiva, SPA y centrada en el usuario</strong> que elimina la fricciГіn burocrГЎtica y agiliza la vida universitaria.</p>

<h2>рҹ§© Patrones de Arquitectura Core</h2>

<h3>1. Arquitectura de Componentes Standalone</h3>
<p>La aplicaciГіn utiliza el patrГіn de componentes standalone de Angular, eliminando la necesidad de NgModules. Cada componente declara sus propias dependencias directamente.</p>
<ul>
    <li><strong>Beneficios:</strong> Menor tamaГұo de bundle (tree-shaking), gestiГіn de dependencias simple y mejor organizaciГіn.</li>
</ul>

<h4>рҹ§© AnatomГӯa de un Componente Standard</h4>
<p>Cada componente se organiza en su propia carpeta con una estructura de archivos predecible:</p>

<pre><code>mi-componente/
в”ңв”Җв”Җ mi-componente.component.ts      # LГіgica y definiciГіn Standalone
в”ңв”Җв”Җ mi-componente.component.html    # Template (HTML5 SemГЎntico)
в”ңв”Җв”Җ mi-componente.component.spec.ts # Tests Unitarios (Jasmine)
в””в”Җв”Җ styles/
    в””в”Җв”Җ mi-componente.component.css # Estilos Scoped (CSS)
</code></pre>

<p><strong>Reglas de oro:</strong></p>
<ul>
    <li>Uso obligatorio de <strong>Signals</strong> para el estado reactivo.</li>
    <li>Encapsulamiento estricto de estilos en la subcarpeta <code>styles/</code>.</li>
    <li>Arquitectura 100% <strong>Standalone</strong> (sin NgModules).</li>
</ul>

<h3>2. Role-Based Access Control (RBAC)</h3>
<p>El sistema implementa tres roles de usuario distintos con capacidades diferenciadas:</p>
<ul>
    <li><strong>ESTUDIANTE:</strong> Inscripciones, consultas acadГ©micas, correlativas.</li>
    <li><strong>PROFESOR:</strong> GestiГіn de materias, carga de notas, mesas de examen.</li>
    <li><strong>ADMIN:</strong> AdministraciГіn total del sistema y usuarios.</li>
</ul>

<h3>3. Flujo de AutenticaciГіn JWT</h3>
<p>La autenticaciГіn se maneja mediante tokens JWT con inyecciГіn automГЎtica vГӯa <strong>Axios Interceptors</strong> (`axiosClient.ts`). El estado de autenticaciГіn se gestiona reactivamente con `BehaviorSubject` en `auth.service.ts`.</p>

<h3>4. Estrategia de Lazy Loading</h3>
<p>Todas las rutas de caracterГӯsticas (features) utilizan `loadComponent` para optimizar la carga inicial, descargando el cГіdigo solo cuando el usuario navega a la ruta concerniente.</p>

<h3>5. Clean Code & Standalone Philosophy</h3>
<p>El proyecto ha sido refactorizado para seguir una arquitectura limpia y moderna:</p>
<ul>
    <li><strong>Sin Comentarios:</strong> El cГіdigo estГЎ diseГұado para ser auto-explicativo mediante nombres claros de variables y funciones, eliminando comentarios innecesarios para un mantenimiento mГЎs ГЎgil.</li>
    <li><strong>Standalone Components:</strong> Uso nativo de componentes sin mГіdulos para reducir el acoplamiento y mejorar el tree-shaking.</li>
</ul>

<h2>рҹҺЁ Sistema de Temas (Theming)</h2>

<h3>вҡ–пёҸ FilosofГӯa de Testeo 1:1</h3>

<p>El proyecto sigue una polГӯtica estricta de <strong>aparejamiento 1:1</strong> entre archivos de cГіdigo y archivos de prueba:</p>
<ul>
    <li><strong>Regla de Oro:</strong> Todo archivo <code>.ts</code> que contenga lГіgica de negocio, servicios o componentes <strong>DEBE</strong> tener su correspondiente archivo <code>.spec.ts</code>.</li>
    <li><strong>UbicaciГіn:</strong> El test debe residir en el mismo directorio que el archivo testeado.</li>
    <li><strong>Integridad:</strong> Esta convenciГіn asegura la estabilidad del sistema a largo plazo y la validaciГіn continua en CI.</li>
</ul>
